# BOM → KiCad Export — Design Document

**Status:** Validated (brainstorm complete) — ready for writing-plans → subagent-driven-development.
**Date:** 2026-06-04

**Goal:** From a revision's frozen BOM and its curated, verified part assets, generate a KiCad-ready project tree (libraries + scaffold + auto-wired power/ground rails), downloadable as a `.zip`, so an L1 learner opens a project that has the tedious setup already done and only the signal wiring + layout left to learn.

**Audience (v1):** the L1 learner. The export's output is a *starting point* they build on in KiCad, not a finished design.

**Tech stack:** Next.js 16 (server action / route handler), TypeScript, Prisma 7 + Neon, Cloudflare R2 (asset blobs in, export blob out via the existing `Artifact` model), KiCad **10** S-expression / JSON file formats. No KiCad binary — output is generated by string-templating the file formats server-side.

---

## 1. What it produces

A `.zip` written to R2 and attached to the revision as a `BOM_EXPORT` artifact (the slot already exists in `ArtifactSubkind`), downloadable from the revision page:

```
<project-slug>/
  <project>.kicad_pro        # JSON: net classes (Power/Ground), design rules, layer stackup, 3D search paths
  <project>.kicad_sch        # parts placed; GND/+3V3/+5V wired via power ports; signal pins left open
  <project>.kicad_pcb        # board setup only (layers/rules); footprints arrive via "Update PCB from Schematic"
  sym-lib-table              # → ${KIPRJMOD}/libs/<project>.kicad_sym
  fp-lib-table               # → ${KIPRJMOD}/libs/<project>.pretty
  libs/
    <project>.kicad_sym      # every symbol, Footprint field pre-set to <nick>:<fp>
    <project>.pretty/        # every footprint (.kicad_mod), 3D model path pre-set
    3dmodels/                # .step / .wrl / .glb models
  bom.csv
  EXPORT_REPORT.md           # asset coverage: verified vs stubbed, per part
```

**Schematic is the source of nets** (KiCad's canonical flow). The PCB ships as board-setup-only; the learner runs *Update PCB from Schematic* to pull footprints + nets across — which both avoids sch/pcb netlist-sync bugs and teaches the standard workflow.

## 2. Data model additions

Nets become **first-class revision data** (not a throwaway config) — queryable, auditable, and the seed of the eventual full netlist. v1 populates GROUND + POWER; `SIGNAL` exists but stays empty until a later phase.

```prisma
model Net {
  id         String    @id @default(cuid())
  revisionId String
  revision   Revision  @relation(fields: [revisionId], references: [id], onDelete: Cascade)
  name       String    // "GND", "+3V3", "+5V", "VBUS"
  netClass   NetClass  // GROUND | POWER | SIGNAL
  trust      FactTrust @default(UNVERIFIED)  // reuse the existing verify gate
  verifiedById String?
  verifiedAt   DateTime?
  createdById  String
  createdBy    User    @relation(fields: [createdById], references: [id], onDelete: Restrict)
  createdAt  DateTime  @default(now())
  updatedAt  DateTime  @updatedAt
  nodes      NetNode[]
  @@unique([revisionId, name])
  @@index([revisionId])
}

model NetNode {
  id     String @id @default(cuid())
  netId  String
  net    Net    @relation(fields: [netId], references: [id], onDelete: Cascade)
  refDes String // one physical designator: "U2", "C2" (NOT the comma-joined BomLine ref)
  pin    String // "VOUT" | "1" — matches the part's PINOUT fact pin number/name
  @@unique([netId, refDes, pin])
  @@index([netId])
}

enum NetClass { GROUND POWER SIGNAL }
```

Notes:
- `Revision` gains `nets Net[]`.
- `NetNode.refDes` is a single designator (e.g. `C2`), even though `BomLine.refDes` groups (`C2,C3,C7`). The export resolves `refDes → BomLine/Part` by membership; this decouples connectivity from BOM grouping and keeps it pin-accurate.
- `NetNode.pin` is validated against the part's PINOUT fact when present (pin number or name).
- Nets carry `trust`; the export wires **verified** nets (configurable to include unverified for a draft export).

## 3. The export pipeline (server-side)

A single server action `exportKicad(revisionId)` (triggered from the revision page) runs the stages below, streams/writes a `.zip` to R2, and records the `BOM_EXPORT` artifact. Pure text/zip generation — no KiCad install.

### 3.1 Library assembly + association
- Pull each BOM part's `PartAsset` triple (SYMBOL/FOOTPRINT/MODEL_3D) from R2.
- Merge symbols into one `<project>.kicad_sym`; copy footprints into `<project>.pretty/`; copy 3D models into `3dmodels/`.
- **Pre-wire associations** (the unique win — normally manual in KiCad): set each symbol's `Footprint` property to `<nick>:<fpName>`, and rewrite each `.kicad_mod`'s `(model ...)` path to the bundled 3D file. Foundry already pairs these per part (`@@unique([partId, kind])`), so we hand them over connected.

### 3.2 Project + board config
- `<project>.kicad_pro` from a per-revision **board config** (layer count/stackup, design rules, net classes incl. a wider Power class, 3D search path). v1 board config can be a sensible default (2-layer, 1oz, learner-friendly clearances) with a few overridable fields.
- `sym-lib-table` / `fp-lib-table` point at the project-local libs via `${KIPRJMOD}`.

### 3.3 Placement
- Auto-place symbols in a grid on the schematic sheet and (post Update-PCB) footprints in a grid — functional, ref-des annotated, non-overlapping. Not a human layout; a usable starting canvas.

### 3.4 Power/ground rail wiring — the crux (highest risk)
- KiCad schematics connect **geometrically**: a pin is "wired" by placing a power-port symbol (or global label) at that pin's absolute coordinate.
- For each `NetNode` (verified GROUND/POWER nets): compute the target pin's absolute endpoint = transform(pin offset+length+angle from the `.kicad_sym` definition, by the symbol instance's `(at x y rot)`), then emit a `power:GND` / `power:+3V3` port (or `(global_label)`) anchored there, with a short connecting wire if needed.
- This requires parsing symbol pin geometry out of the S-expr. **Build the library/scaffold foundation first; layer this last.**

### 3.5 Missing assets → graceful stubs + coverage report
- 16 of 17 BOM parts have no assets today. For a missing SYMBOL, synthesize a **stub symbol from the PINOUT fact** (a box with the real pins labeled by number/name/type) where a pinout exists, else a generic box. For a missing FOOTPRINT, emit a generic footprint matching the `footprint`/package string. Missing 3D → omit.
- `EXPORT_REPORT.md` lists, per part: symbol/footprint/3D = verified | unverified | stubbed | missing. This is the precise asset checklist the export-first approach buys us, and it surfaces any `PartAsset` field the export turns out to need.

## 4. Net editor UI

A proper UI (not JSON) on the revision, since nets are real data:
- **"Derive rails"** action: scans BOM parts' PINOUT facts → creates the `GND` net + a `NetNode` for every `gnd`-type pin across all designators; **proposes** POWER nets from `power`-type pins (UNVERIFIED), leaving the human to resolve ambiguity (e.g. assign the LDO's `VOUT → +3V3`, `VIN → +5V`).
- A **table** of nets → their nodes (refDes.pin), with add/remove and the standard UNVERIFIED → VERIFIED verify buttons (mirrors the facts/assets surface).
- Net assignment edits use the same optimistic-concurrency + verify-gate patterns already in the codebase.

---

## 5. Format + workflow decisions (locked)
- **Target KiCad 10** file formats.
- **Schematic is the source of nets.**
- PCB ships board-setup-only; learner runs *Update PCB from Schematic* (robust + teaches the flow).

---

## 6. Build sequencing
Foundation before the risky bit:
1. Schema: `Net`/`NetNode`/`NetClass` migration + actions + tests.
2. Net editor UI (derive + table + verify).
3. Export engine: library assembly + associations + lib-tables (testable against U1's real assets).
4. Project/board config (`.kicad_pro`) + placement + minimal `.kicad_pcb`.
5. Stub generation + `EXPORT_REPORT.md` coverage.
6. Power/ground rail geometric wiring in `.kicad_sch` (the crux), reading verified nets.
7. Zip assembly + R2 write + `BOM_EXPORT` artifact + revision-page download button.

---

## 7. Testing strategy
- Assets are sparse (only U1), so: unit-test S-expr/JSON generators against **synthetic fixtures**; integration-test the full pipeline against **U1's real assets** (the real-asset path) + the **stub path** (missing-asset parts).
- Golden-file tests for generated `.kicad_sym` / `.kicad_sch` / `.kicad_pro` snippets.
- A round-trip sanity check: the generated files should open in KiCad 10 (manual acceptance once a real-ish board exists; automated structural validation in CI).
- Pin-geometry math (3.4) gets focused unit tests (placement at known rotations).

---

## 8. Risks & open questions
- **Geometric schematic wiring (3.4)** is the hardest, highest-uncertainty piece — isolated and sequenced last so the rest ships regardless.
- **KiCad 10 format drift** — confirm exact S-expr/JSON shapes against a KiCad 10 reference project early (generate one by hand, diff our output).
- **Pinout pin identity** — `NetNode.pin` must match what the symbol/PINOUT calls the pin (number vs name); needs a consistent convention + validation.
- **Board config surface** — v1 uses defaults; how much to expose is a follow-up.

---

## 9. Out of scope (future phases)
- Full **signal-net** capture (the `SIGNAL` class) — the model supports it; UI/derivation deferred.
- Auto-**routing** and pretty schematic/PCB layout (that's the learner's/engineer's work).
- A full **project scaffold** beyond board-setup (extra sheets, hierarchical design).
- **CAD-asset auto-fetch** (SnapEDA/SamacSys APIs) — separate effort to fill the 16-empty-parts gap; the coverage report from §3.5 is its checklist.
